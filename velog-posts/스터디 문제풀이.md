<h1 id="10-예외처리">10. 예외처리</h1>
<h2 id="문제-1-다음-중-일반-예외checked-exception에-대한-설명으로-가장-적절한-것은"><strong>문제 1. 다음 중 ‘일반 예외(checked exception)’에 대한 설명으로 가장 적절한 것은?</strong></h2>
<ol>
<li>실행 중 갑자기 발생하며 컴파일 시 검사되지 않는다.</li>
<li>RuntimeException을 상속하는 예외들이다.</li>
<li>컴파일 시 예외 처리 여부를 검사하며, 처리하지 않으면 컴파일 오류가 발생한다.</li>
<li>반드시 프로그램을 종료시키는 예외이다.</li>
</ol>
<p>정답: 3. 컴파일 시 예외 처리 여부를 검사하며, 처리하지 않으면 컴파일 오류가 발생한다.
설명: <em>일반 예외(checked exception)</em>는 컴파일러가 메서드에서 처리(try-catch)하거나 throws로 선언했는지 검사한다. RuntimeException 계열은 비검사 예외(unchecked) 이다.</p>
<h2 id="문제-2-다음-중-try-catch-블록-작성-시-올바른-순서-규칙은-무엇인가"><strong>문제 2. 다음 중 try-catch 블록 작성 시 올바른 순서 규칙은 무엇인가?</strong></h2>
<ol>
<li>상위 예외 클래스의 catch를 먼저, 하위 예외를 뒤에 둔다.</li>
<li>하위 예외 클래스의 catch를 먼저, 상위 예외를 뒤에 둔다.</li>
<li>순서는 상관없다.</li>
<li>catch를 여러 개 쓰려면 모두 같은 예외 타입이어야 한다.</li>
</ol>
<p>정답: 2. 하위 예외 클래스의 catch를 먼저, 상위 예외를 뒤에 둔다.
설명: 상위 예외(예: Exception)를 먼저 쓰면 더 구체적인 하위 예외 catch에 도달하지 못하므로 컴파일 오류가 난다.</p>
<h2 id="문제-3-다음-중-예외-떠넘기기throws에-대한-설명으로-가장-적절한-것은"><strong>문제 3. 다음 중 ‘예외 떠넘기기(throws)’에 대한 설명으로 가장 적절한 것은?</strong></h2>
<ol>
<li>throws는 실행 예외만 떠넘길 수 있다.</li>
<li>throws를 사용한 메소드는 내부에서 try-catch를 사용할 수 없다.</li>
<li>throws는 메소드 호출자에게 예외 처리를 맡기겠다는 의미이다.</li>
<li>throws는 예외를 즉시 처리하고 프로그램 실행을 계속하는 기능이다.</li>
</ol>
<p>정답: 3. throws는 메소드 호출자에게 예외 처리를 맡기겠다는 의미이다.
설명: throws는 해당 메서드가 예외를 직접 처리하지 않고 호출자에게 전달한다는 선언이다. throws를 써도 그 메서드 내부에서 try-catch를 사용할 수 있다.</p>

<h1 id="13-컬렉션-프레임워크">13. 컬렉션 프레임워크</h1>
<p><strong>문제 1 (객관식 — 기본 특성)</strong></p>
<p>다음 중 <strong>List</strong>와 <strong>Set</strong>의 차이로 <strong>옳지 않은 것</strong>은?</p>
<p>A. List는 요소의 순서(index)를 유지하지만 Set은 일반적으로 순서를 보장하지 않는다.
B. List는 중복 요소를 허용하고 Set은 중복을 허용하지 않는다.
C. Set은 항상 정렬된 상태를 유지한다.
D. List는 get(int index)로 요소를 직접 참조할 수 있다.</p>
<p>정답: C. Set은 항상 정렬된 상태를 유지한다. (옳지 않음)
설명: Set 구현체 중 TreeSet은 정렬, LinkedHashSet은 삽입 순서 유지, HashSet은 순서 보장하지 않음. 따라서 “항상 정렬”은 틀리다.</p>
<p><strong>문제 2 (객관식 — Set 구현체 차이)</strong></p>
<p>HashSet, LinkedHashSet, TreeSet의 특징으로 <strong>옳게 짝지어진 것</strong>은?(O/X)</p>
<ol>
<li>HashSet — 해시 기반, 순서 보장 X, null 허용</li>
<li>LinkedHashSet — 해시+링크로 삽입 순서 유지, 반복자 순서 예측 가능</li>
<li>TreeSet — 정렬(자연정렬 또는 Comparator), null 삽입 시 NullPointerException 발생(대부분의 경우)</li>
</ol>
<p>HashSet — 해시 기반, 순서 보장 X, null 허용. → O
LinkedHashSet — 해시 + 링크로 삽입 순서 유지, 반복자 순서 예측 가능. → O
TreeSet — 정렬(자연정렬 또는 Comparator), null 삽입 시 NullPointerException 발생(대부분의 경우). → O
설명: 모두 올바른 설명이다 (TreeSet은 비교 연산을 필요로 하므로 null과 혼용하면 NullPointerException 발생 가능).</p>
<p><strong>문제 3 (객관식 — Map 특징)</strong></p>
<p>다음 설명 중 HashMap과 Hashtable에 대한 <strong>옳은</strong> 설명은?</p>
<p>A. HashMap은 동기화(synchronized)되어 있고 Hashtable은 동기화되지 않았다.
B. HashMap은 null 키와 null 값 둘 다 허용하지만 Hashtable은 null 키/값을 허용하지 않는다.
C. Hashtable은 성능이 항상 HashMap보다 빠르다.
D. HashMap은 삽입 순서를 보장한다.</p>
<p>정답: B. HashMap은 null 키와 null 값 둘 다 허용하지만 Hashtable은 null 키/값을 허용하지 않는다.
설명: Hashtable은 레거시로 동기화된 클래스이고 null 키/값을 허용하지 않는다. HashMap은 null 키(하나)와 null 값들을 허용한다. (A, C, D는 모두 틀림: A는 반대, C는 성능 보장 불가, D는 HashMap은 삽입 순서 보장하지 않음.)</p>

<p><strong>문제 5 (객관식 — Queue 메소드 차이)</strong></p>
<p>Queue 인터페이스 관련해서 다음 메소드 쌍의 차이로 <strong>옳은 것</strong>은?</p>
<ul>
<li>add(e) vs offer(e)</li>
<li>remove() vs poll()</li>
<li>element() vs peek()</li>
</ul>
<p>A. add/offer: 둘 다 동일하게 동작(예외 처리 없음)
B. add(e)는 큐가 꽉 차면 IllegalStateException을 던지고, offer(e)는 false를 반환한다.
C. remove()는 요소가 없을 때 null을 반환하고 poll()은 예외를 던진다.
D. element()는 요소가 없을 때 null 반환, peek()는 예외를 던진다.</p>
<p>정답: B. add(e)는 큐가 꽉 차면 IllegalStateException을 던지고, offer(e)는 false를 반환한다.
설명 (정리):
add(e) vs offer(e)
add(e)는 용량 제한으로 삽입 불가 시 예외(IllegalStateException) 발생.
offer(e)는 삽입 불가 시 false 반환.
remove() vs poll()
remove()는 요소가 없으면 NoSuchElementException 던짐.
poll()은 요소가 없으면 null 반환.
element() vs peek()
element()는 요소가 없으면 NoSuchElementException 던짐.
peek()은 요소가 없으면 null 반환.
따라서 B가 맞고 A,C,D는 모두 거짓(설명 반대 또는 잘못된 내용).</p>
